# Substantive Review

Game Studio treats scripts as repository hygiene and Codex as the reviewer of
game substance.

## Principle

Automated checks are useful, but they are not game direction. A project advances
when the right roles can defend the playable prototype and the internal artifacts
that support it.

## Codex-Led Review Surfaces

- `core/references/game-craft-source-map.md`: source-backed craft vocabulary.
- `core/references/codex-review-practice.md`: review method and verdict shape.
- `core/roles/playbooks/`: discipline-specific review behavior.
- `core/rubrics/`: readiness criteria for direction, production, playtest, accessibility, and moment-to-moment quality.
- `core/templates/`: project artifacts that force proof targets, internal
  evidence artifacts, tradeoffs, and follow-up decisions.
- `research/`: source reports used to improve this framework.

## Required Review Shapes

### Concept Playability Review

Before Direction Lock, review whether the concept has a player fantasy, first
verb, player-caused change, first-minute learning, failure condition, smallest
playable prototype, and rejection trigger. Do not approve a concept because it
sounds exciting.

### First Playable Review

After the first executable prototype exists, review the loop:

```text
Verb -> read -> risk -> response -> learning -> next choice
```

The review should cite captures, QA notes, playtest notes, or bug triage entries
and then decide whether to advance, repeat the prototype, narrow scope, park the
concept, or reject it.

### Release Validation Review

Before public surfaces change, map every public claim to a concrete internal
artifact. Claims without build truth should be revised, removed, or blocked.

## What To Avoid

- Do not treat a filled template as a good plan.
- Do not treat passing commands as playable quality.
- Do not let role names replace role judgment.
- Do not approve public claims without build truth and concrete internal
  artifacts.
- Do not use vague confidence in place of a review verdict.
- Do not add more process when a sharper review question would solve the problem.

## Minimal Script Policy

Use scripts only for cheap invariants. For game quality, require a written role
review that names the proof target, proof checked, internal artifacts checked,
concerns, tradeoffs, and the next prototype.

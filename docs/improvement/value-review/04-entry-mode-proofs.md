# Entry Mode Proofs

The product promise depends on all three entry modes working. The repository
currently demonstrates taste-to-game best. It needs parallel proof for existing
direction and brownfield unknown projects.

## Entry Modes To Prove

| Entry Mode | Current Coverage | Missing Proof |
| --- | --- | --- |
| `taste-to-game` | Strong sample project with concept slate, proof budget, and `CONCERNS` review. | A real installed project with a playable artifact. |
| `direction-carry-in` | Templates and route rules exist. | Filled example showing preservation before improvement. |
| `brownfield-unknown` | Templates and route rules exist. | Filled example showing audit before new direction. |

## Example Set

Add:

```text
examples/sample-direction-carry-in/
examples/sample-brownfield-unknown/
examples/sample-playable-proof/
```

## Direction Carry-In Example

This example should start with an existing game idea, pitch, or prototype.

Required artifacts:

- source direction inventory
- authority owner
- protected direction
- missing proof audit
- current stage inference
- first proof target
- decision record for any proposed change

The example should prove that Game Studio improves the project without
overwriting the carried direction.

## Brownfield Unknown Example

This example should start with a messy repo or mixed notes.

Required artifacts:

- existing material inventory
- stale or conflicting promises
- playable truth versus written claims
- protected files
- authority uncertainty
- recommended route
- `USER_DECISION_NEEDED` stop condition if ownership is unclear

The example should prove that Game Studio audits before inventing new direction.

## Playable Proof Example

At least one example should include a tiny runnable game or inspectable playable
artifact. It can be browser-based and intentionally small.

Required artifacts:

- build or start command
- first-minute capture or capture note
- playtest note
- council or first-playable review
- next proof after `CONCERNS`

The example does not need production polish. It needs to make the proof loop
real.

## Proof Of Improvement

A reviewer should be able to choose any entry mode and find:

- the starting context
- the first artifact
- the human authority boundary
- the proof target
- the review result
- the next proof or stop condition

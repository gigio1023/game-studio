# Accessibility Rubric

Use this rubric from prototype onward when reviewing player-facing input, UI,
audio, text, motion, difficulty, or public feature claims.

## Barrier Categories

| Category | Review focus |
| --- | --- |
| Motor | Remapping, alternative input, hold/tap options, timing pressure, menu navigation. |
| Vision | Text size, contrast, color independence, object clarity, UI focus, screen motion. |
| Hearing | Subtitles, captions, visual cues, separate volume controls, non-audio warnings. |
| Cognitive | Clear goals, readable instructions, difficulty options, memory load, error recovery. |
| Speech | Voice alternatives, text alternatives, communication settings. |
| General | Accessible documentation, feature disclosure, support, and known limitations. |

## Verdicts

- `READY`: Required access paths are implemented or honestly out of scope for the current stage.
- `CONCERNS`: Access barriers exist but are documented, bounded, and assigned.
- `NOT_READY`: A core path blocks players without disclosure, workaround, or plan.

## Internal Evidence Artifacts

- Settings screenshot or capture.
- Input-device notes.
- Subtitle or caption capture.
- Color and contrast review notes.
- Motion comfort notes.
- Player or expert accessibility feedback.
- Release claim mapping.

## Release Claim Impact

If an accessibility feature is public-facing, the review must map the claim to a
concrete internal evidence artifact. If the support is partial, unavailable, or
out of scope for the current stage, disclose the limitation or remove the claim.
